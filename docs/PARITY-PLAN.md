## 1. Goal restatement

**"Fork Jito Restaking to complete feature parity with the EigenLayer EVM features this system depends on" does NOT mean a cold fork of jito-foundation/restaking.** The recon shows why:

- Jito Restaking (RestkWeAVL8..., Vau1t6sLNx...) is an audited (Ottersec/Offside/Certora), mainnet-live, actively maintained protocol whose account model (NCN/Operator/Vault, ticket handshakes, SlotToggle epochs, delegation, withdrawal tickets) already covers the registration/staking/delegation surface Gas Killer consumes from EigenLayer [jito recon]. NCNs are built as **separate programs on top** — that is the canonical pattern (ncn-template, jito-tip-router) [jito recon]. The ck/ncn draft already proves the NCN side speaks to **unmodified** jito restaking + vault programs ("no fork of Jito itself needed; parity work is NCN-side only") [existingIntegration].
- The on-chain SDK consumes **exactly one EigenLayer runtime primitive**: `IBLSSignatureChecker.checkSignatures` inside `verifyAndUpdate` (src/GasKillerSDK.sol:72-73); everything else is SDK-owned logic (transition nonce, sha256 digest, staleness window, 66% threshold, diff applier) or off-chain (EigenStakingClient bootstrap, BLSApkRegistry bindings, getNonSignerStakesAndSignature retrieval, Submitter) [eigenlayerDeps].
- Two independent drafts of the Solana/Jito BLS NCN already exist: Unboxed `ncn-program-template-bls` (Aug 2025, more complete on-chain + test harness) and jito-foundation `jito-bls-ncn` ck/ncn (Oct 2025, better stake model + commonware adapter concepts) [GH]. The commonware-restaking chassis has since been refactored into exactly the pluggable shape (certificate::Scheme + CertIndex + scheme-specific Submitter, consumed via CertifiedHeight\<S\>) that PR #121's closure asked for [GH].

**Therefore the goal splits into two layers:**

- **Layer 1 (primary scope): the Jito/Solana backend of commonware-restaking reaches parity with its EigenLayer backend and with the settlement primitive Gas Killer's consumer contracts require.** Concretely, the 8-point checklist derived from chassis consumption [existingIntegration]: (1) operator BLS-key + socket registry; (2) stake state at a reference point; (3) on-chain aggregate verification with non-signer subtraction; (4) threshold read from chain; (5) registration gated by restaking opt-in; (6) e2e local stack + CI; (7) key rotation; (8) ejection/churn + slashing hooks — plus the Gas Killer settlement primitive (verifyAndUpdate analog: replay nonce, digest binding, single-commitment write, events-as-data) and the ops envelope (docker-compose one-command stack, green CI). Items 1–5 exist in draft form; 6–8 and the settlement consumer are the real build.
  **Scope honesty on the Layer 1 claim:** Layer 1 makes "everything works e2e in CI" literally true for the **commonware-restaking toolkit stack on Solana** (chassis + Jito backend + counter example + the gaskiller-settlement program with its ArraySummation-equivalent consumer). It does **not** by itself port **gas-killer/service** — which has its own vendored EigenStakingClient, BLSApkRegistry/OperatorStateRetriever bindings, verifyAndUpdate submission path, /trigger + prewarm + bridge APIs, and event listeners [eigenlayerDeps] — nor make gas-analyzer emit Solana-format diffs for real workloads (gas-analyzer produces EVM StateUpdateTypes from revm traces). Two consequences are made explicit: (a) the **EVM-sketch→borsh mapping rule** (how an EVM single-SSTORE sketch becomes the borsh {Store,Event} diff the settlement program consumes — trivial for single-slot consumers, where the new root *is* the commitment value, but it must be stated and tested) is a named **Phase 4 deliverable**; (b) the **service submitter/listener/API port** is a separately-scoped follow-on effort, surfaced as blocking decision #8 in §6 — it is not silently included in the 8-week estimate.
- **Layer 2 (roadmap scope): the broader EigenLayer core+middleware checklist (32 items, v1.13.0 core + middleware v1.5.0 [eigenlayer recon]) mapped onto Jito restaking as a prioritized backlog** — slashing execution, redistribution, rewards, operator sets/unique stake, multichain, UAM, etc. Most of this is **not consumed** by Gas Killer today ("AVSDirectory, DelegationManager, AllocationManager, RewardsCoordinator, ServiceManagerBase, slashers... are all present in lib/ but never invoked" [eigenlayerDeps]; "no slashing implemented in the live system... no reward distribution mentioned anywhere" [transcriptB]).

**Recommended scope split: commit to Layer 1 as the deliverable; treat Layer 2 as a prioritized roadmap with only slashing-hooks and rewards-routing designed (not built) in this effort.** Rationale: Layer 1 is the toolkit-side foundation everything else plugs into; Layer 2 chases a target that is itself mid-migration and partially unaudited on Ethereum ("middleware contracts have not yet been fully audited... migration to operator sets is gradual" [eigenlayer recon item 20]), and building it ahead of consumption would repeat the vendored-but-unused pattern the EVM repo already exhibits.

## 3. Architecture of the revival

**Primary path (per [GH]): revive the existing drafts as a first-class Jito/Solana backend inside/beside the refactored commonware-restaking chassis — additive, zero deletions of the EVM path.**

### 3.1 Per-layer base choice

**On-chain NCN program layer: Unboxed `ncn-program-template-bls` as the base, with jbn donations.**
Rationale:
- Licensing: Unboxed is Apache-2.0/MIT [GH]; **jito-bls-ncn has NO license** [jito recon] — direct code reuse from ck/ncn is legally blocked without Jito Foundation clearance (see §6).
- Registration crypto strictly stronger: real on-chain proof-of-possession vs jbn's pair-consistency-only check ("permissionless-operator-ready" vs "must stay admin-gated") [existingIntegration].
- Engineering shape: borsh + shank IDL + generated Rust/JS clients + real jito_bytemuck/jito_restaking_core/jito_vault_core crates, vs jbn's hand-rolled repr(C,packed) mirrors, duplicate-discriminator bug (0x05/0x12), `todo!()` RemoveBlsOperator, `.expect()` on-chain panic paths, array-contiguity assumption [existingIntegration].
- Testing: 18 integration modules incl. fuzz + simulation vs real .so fixtures, TestBuilder, keeper loop — best-in-class of the two [existingIntegration].
- **Required changes to the Unboxed base** (the jbn donations, re-implemented not copied, given the license issue): (a) restore multi-vault stake weighting (MAX_VAULTS=1 → jbn's per-vault weight_bps RollingSnapshot design as the donor concept); (b) replace count-based quorum (fail if non_signers > registered/3) with stake-weighted BPS threshold from Config; (c) replace counter-as-message with digest-as-message and **make the verify path stateless** (§3.4); (d) epoch-decay of stale operator snapshots (jbn concept); (e) implement operator removal/ejection **with tombstoned indices, snapshot-generation binding, and atomic APK reconciliation** (2a ejection row — rated M, not S: both drafts' failure to ship it is evidence of the difficulty).

**BLS math layer: Unboxed's pluggable-scheme structure (`core/src/schemes/` traits, g1/g2 point types), with the crypto corrected to match eigenlayer-middleware exactly — neither draft's alpha survives.**
Rationale: the recon confirms **three mutually incompatible variants** of `compute_alpha`: jbn returns raw SHA256 with no reduction; Unboxed reduces mod **Fq (base field)** not Fr; eigenlayer-middleware BLSSignatureChecker.sol#L214 computes gamma = **keccak256**(msgHash ‖ apk.X ‖ apk.Y ‖ apkG2 ‖ sigma) reduced mod **FR_MODULUS** [existingIntegration, eigenlayerDeps]. Both hash-to-curve implementations are the same try-and-increment family and both carry the eigenlayer-middleware #172 exposure [existingIntegration]. Required work items (apply to the chosen implementation, mandated by ron-review [GH]):

- **Work item CRYPTO-1 (compute_alpha, EigenLayer-exact):** redefine alpha as **keccak256 over EigenLayer's exact preimage and ordering (msgHash, apk.X, apk.Y, apkG2.X/Y, sigma.X, sigma.Y), reduced modulo the BN254 group order Fr** — matching BLSSignatureChecker.sol#L214 byte-for-byte. Solana has a keccak256 syscall (85 CU base + 1 CU/byte [solana]), so this costs nothing material and makes the differential gate against the Solidity reference actually satisfiable (a SHA256-based alpha could never equal keccak-based gamma; the original criterion was unsatisfiable as written). *Acceptance criteria:* (i) unit test vectors where the raw hash ≥ Fr demonstrating pre/post-fix divergence; (ii) cross-implementation differential test: the on-chain program, the off-chain aggregator, an independent ark-bn254 reference, and **eigenlayer-middleware's Solidity BLSSignatureChecker (via the EVM test harness)** produce identical alpha and identical verify outcomes on ≥1000 randomized cases; (iii) explicit test that an oversized scalar fed to the alt_bn128 mul syscall no longer occurs. (Fallback, only if a preimage-layout incompatibility surfaces during implementation: drop EVM-byte-compatibility, respecify (ii) as byte-identical alpha across the *Solana* stack only — program / aggregator / ark-bn254 reference — with a written note that EigenLayer byte-compat is not needed since certificates never cross chains. This fallback requires an explicit sign-off, not a silent gate deletion.)
- **Work item CRYPTO-2 (map_to_curve / issue #172):** enforce **digest-only hashing** — the hash-to-curve input must always be an already-collision-resistant 32-byte digest, never a raw message (this also matches the chassis rule: "sign digest-only — module docs forbid namespace/height in the preimage" [existingIntegration]). *Acceptance criteria:* (i) type-level enforcement (input type is a 32-byte digest newtype produced only by the domain-hash function); (ii) documented domain separation for the settlement digest; (iii) test asserting the program rejects/never exposes a raw-message path; (iv) the in-code #172 TODO is resolved with a written rationale in the crate docs; (v) try-and-increment retains the ≥-modulus rejection (no modulo bias) present in both drafts.
- Additional crypto hygiene from the recon fix-list: PoP required at registration (anti-rogue-key), consensus-not-reached error no longer misreported as `AccountAlreadyInitialized` (jbn), no `.expect()` on untrusted decode paths.

### 3.4 Account model of the BLS NCN program

Base = Unboxed's 9 account types with modifications:

- `Config` (PDA per NCN): admin, epochs_before_stall, epochs_after_consensus_before_close, valid_slots_after_consensus, minimum_stake, **consensus_threshold_bps (new — threshold-from-chain)**, fee params.
- `VaultRegistry` (PDA): **multi-vault restored** — per-vault entry with weight_bps (jbn donor design), MAX_VAULTS raised from 1 (target: match jbn's bound, with the 300-vs-256 array/const mismatch resolved to one constant).
- `NCNOperatorAccount` (PDA per ncn+operator): compressed g1/g2, operator index, ip+port, PoP-verified at registration; `UpdateOperatorBN128Keys` (atomic APK sub/add + generation bump, §2a key-rotation row), `UpdateOperatorIpPort`, **`RemoveOperator` (implemented, ejector-gated; tombstones the index — never reused within an epoch — subtracts the G1 from the running APK, bumps the snapshot generation)**.
- `Snapshot` (PDA, realloc-grown): operators_registered, per-operator StakeWeights accumulated per-vault from real `VaultOperatorDelegation` (empty account = zero stake), running total_aggregated_g1_pubkey (incremental add/sub on register/remove/rotate), last_snapshot_slot, **generation counter (bumped on any operator-set or key change; certificates are valid only against the generation they were assembled under)**, epoch-staleness decay (jbn semantics: staked+enqueued+cooling → staked → 0 by epoch age). Tombstoned entries retain their index slot for bitmap stability; compaction only at epoch rollover.
- **Verification path — `VerifyCertificate{digest: [u8;32], agg_g2, agg_sig, bitmap, expected_generation}`: STATELESS.** Recomputes hash-to-curve over the domain-separated digest, subtracts non-signer G1s, checks stake_bps ≥ Config.consensus_threshold_bps, checks snapshot generation and slot-freshness — and **mutates no NCN state**. This mirrors the EVM analog exactly: `checkSignatures` is a pure view; consumers call it and own their replay protection [eigenlayerDeps]. Exposed both as an instruction (for the standalone demo) and as the CPI-able path consumers invoke (D-VERIFY, §3.5). A stateful NCN-global counter in the verify path (as in both drafts) is rejected: it would serialize all consumers on one PDA write-lock, create undefined cross-consumer ordering, and leave the counter's value unpredictable to any consumer building a digest.
- **Replay protection — RESOLVED design decision D-MSG: message = the raw 32-byte digest; the replay nonce is *consumer-local*.** Each consumer (the settlement program's state PDA, or the demo counter's own PDA) holds its own monotonic `transition_count` — the exact StateTracker analog — and **binds it into the digest preimage** (transition_index is inside the sha256 preimage, mirroring the EVM: "cert binds only the digest... replay across heights is safe due to on-chain transition-index ordering" [eigenlayerDeps]). At the NCN level a certificate verifies any number of times, and that is correct: the NCN is a verifier, not a sequencer; replay safety lives where the state lives, per consumer. `VoteCounter` is repurposed as the **demo consumer's** state (a thin `CastVote` demo instruction = VerifyCertificate + bump its own counter, with the counter value bound into the demo digest) — it is no longer an NCN-global replay nonce and no cross-consumer state is shared. This resolves before Phase 1, since it fixes both the instruction set and the scheme's signature domain simultaneously.
- `AccountPayer` (rent payer PDA, Unboxed pattern).
- Quorum rule: signed stake_bps ≥ Config.consensus_threshold_bps (stake-weighted), with per-operator minimum_stake gate retained.

### 3.5 Gas Killer settlement mapping (unbounded execution → single-commitment settle on Solana)

The EVM flow — off-chain UnboundedV1 (2^40) simulation → quorum signs sha256(abi.encode(transitionIndex, address(this), selector, storageUpdates)) → verifyAndUpdate does one SSTORE + LOGs [transcriptA, eigenlayerDeps] — maps as:

- **Consumer = a per-app `gaskiller-settlement` program + one state PDA** holding: `commitment_root: [u8;32]` (the *_ROOT_SLOT analog — "maps naturally to one Solana account holding a 32-byte commitment" [transcriptA]), `transition_count: u64` (**the StateTracker analog and the D-MSG replay nonce — consumer-local by design, §3.4**), NCN + config references.
- **Digest**: sha256(borsh(transition_index, state_pda_pubkey, instruction_discriminator, state_updates)) — pubkey replaces address(this), 8-byte discriminator replaces bytes4 selector, borsh replaces abi.encode [repoMap "EVM-specific message-hash commitment"]. The transition_index inside the preimage is what makes NCN-level certificate replay harmless.
- **EVM-sketch→borsh mapping rule (named deliverable, resolves the undefined producer for real workloads):** for single-slot consumers the rule is: EVM Store{slot=*_ROOT_SLOT, value=newRoot} → borsh Store{data=newRoot} (the new root *is* the commitment value); EVM LOGn{topics, data} → borsh Event{discriminant=event-signature hash truncation, payload=data}; anything else is gate-excluded. Stated in docs and covered by a differential test (EVM operator-replay payload → mapped borsh diff → identical settled commitment).
- **Settle instruction** (verifyAndUpdate analog): check snapshot slot-freshness (blockStaleMeasure analog); check transition_index + 1 == transition_count; recompute digest and require == certified digest; verify the NCN certificate via the **stateless** verify path (**design decision D-VERIFY**: CPI into the NCN's VerifyCertificate vs inline verify against the Snapshot PDA — decided by CU/CPI-depth measurement in Phase 4; both are pure verifications, so the choice is cost/audit-surface only, with no shared-state coupling either way); apply the diff: exactly one commitment-account write + self-CPI events carrying output data (ChatAnswered analog); increment the consumer's transition_count. Preserve the error surface: InsufficientQuorumThreshold / InvalidTransitionIndex / StaleSlot / FutureSlot / InvalidStorageUpdates equivalents [repoMap].
- **Diff wire format**: borsh enum {Store{data}, Event{discriminant, payload}} — the Solana IStateUpdateTypes; CALL deferred, CREATE excluded (gate-excluded on EVM anyway [repoMap]).
- **Events-as-data mechanism (named Phase 4 deliverable — the toolchain is native no-Anchor, so nothing ships this for us):** a hand-rolled **self-CPI event helper** (program invokes itself with event bytes as instruction data, the non-truncatable channel [solana]) plus a documented **event ABI** (discriminant registry + borsh payload layouts) for Geyser/webhook indexers. Hard cap acknowledged: **CPI instruction data is limited to 10 KiB in Agave (MAX_CPI_INSTRUCTION_DATA_LEN)** — "never truncated" is not "unbounded."
- **Large payloads / DA channel (redesign, flagged in §7):** instruction data ≤ ~1100 bytes per top-level tx [solana]; self-CPI events ≤10 KiB. Multi-KB outputs (LLM answers) therefore live in a **digest-verified buffer account**: staged across multiple untrusted txs, hash-checked against the certified digest at settle time; the settle event carries **digest + buffer pubkey**, not the payload. Buffer rent/close policy: **retain-until-indexed** — the buffer stays open (rent-funded by the requester) until the indexer acks, then close-and-refund; closing at settle time would break later indexing. Chunked ≤10 KiB events are the documented alternative for mid-size payloads.
- **Shape-gate invariant re-expressed**: "exactly one consumer commitment write + counter bump" becomes "settle writes exactly the state PDA (+ event self-CPI)"; enforced by the analyzer off-chain and by tests asserting the account-write set (vm.record analog).
- **Profile/env commitments**: versioned sim-profile id + env_commitment (overlay manifest) recorded in the consumer Config PDA — the "place to pin env/overlay commitments so slashing evidence references the exact execution environment" [transcriptA]; fraud-proof binding itself stays roadmap, matching EVM status ("SP1 guest chainConfigHash binding... the one unimplemented mainnet-blocker" [transcriptA]).
- Second verb `settlePrefix` (prefix-resume anchors [transcriptB]) = a second instruction writing a settledRoots entry, own single-write shape — same pattern, add when the sharded consumers are ported.

## 7. Risks & honesty notes

- **The two soundness flags are confirmed in code, and worse than "two bugs":** there are **three incompatible alpha definitions** across jbn (raw SHA256, no reduction), Unboxed (SHA256 mod Fq — wrong hash *and* wrong modulus vs EigenLayer), and EigenLayer (keccak256 mod Fr) [existingIntegration, eigenlayerDeps]. CRYPTO-1 resolves this by adopting EigenLayer's exact keccak preimage + mod-Fr reduction (satisfiable on Solana via the keccak syscall), making the Solidity differential gate real rather than aspirational; until it lands with differential vectors, no signature produced by one stack verifies reliably against another. The alt_bn128 syscall "tolerating oversized scalars" means the bug is silent, not loud.
- **Native-ETH restaking has no Solana analog.** EigenPods/beacon-proof checkpointing is Ethereum-specific; parity is "different mechanism, equivalent capability" via SPL-token/LST vaults — this must be stated, not papered over [eigenlayer item 4, jito].
- **Executable slashing is impossible-analog on current Jito v1:** the deployed vault program has **no Slash instruction** — only registration/ticket plumbing [jito]. Full EigenLayer slashing parity (AllocationManager wads, redistribution, 7-day resolution) is blocked on Jito upstream. Honest mitigation: Gas Killer's live EVM system also has no slashing ("security is redundant k-execution + byte-equality + stake-weighted quorum" [transcriptB]), so Layer 1 parity is real parity with what exists. Relatedly, EigenLayer's independent per-delay safety lattice (item 19) collapses onto Jito's single `epoch_length` knob — a structural limitation documented in the §2b row.
- **Gas semantics don't port; they mostly don't need to.** UnboundedV1 (2^40) is an *off-chain simulation* constant — chain-agnostic. But the Solana settlement side has hard constraints the EVM side lacks: 1232-byte tx (≈1100B ix data), **10 KiB CPI instruction data (caps self-CPI events)**, 1.4M CU/tx, 64MiB loaded-account data [solana]. Consequences: (a) multi-KB events-as-data payloads move to the digest-verified buffer-account channel (event = digest + buffer pubkey; retain-until-indexed close policy) — a real protocol change vs the EVM one-tx settle; (b) events are non-consensus at the log level — the self-CPI helper + indexer is a weaker trustlessness story than eth_getLogs, and the helper itself is our own native code (no Anchor); (c) the pairing (~250k CU claimed for jbn Vote [existingIntegration]) fits, but D-VERIFY (CPI vs inline) needs measurement, not assumption.
- **Operator removal/rotation is genuinely hard, not an S-ticket:** jbn shipped removal as `todo!()` and Unboxed omitted it entirely — the incremental APK, bitmap index stability, and frozen-set engine make it an atomic multi-invariant change (§2a). The tombstone + generation-bound design is specified, but a wrong index-reuse rule fails *silently* (one operator's signature slot attributed to another), which is why the between-assembly-and-submission test is a Phase 1 exit criterion.
- **BN254 vs BLS12-381 status:** BN254 G1 + pairing syscalls are mainnet-live (sufficient — both drafts run on them); native G2 arithmetic (SIMD-0302) is only in Review, and BLS12-381 syscalls (SIMD-0388) are devnet-only, mainnet ~Q3 2026 [solana]. ELIP-007 (BLS12-381 for EigenLayer, authored by the user) is still Draft with nothing live [eigenlayer item 31] — do not couple this plan to either curve migration.
- **Upstream-divergence risk:** ck/ncn (Oct 2025) and both Unboxed branches predate the chassis refactor (PRs #184/#185/#186, active through 2026-07-14 [GH]); mohammed-changes is 183 commits behind its own upstream dev. Anything salvaged is reference code, and the additive PR must track a moving chassis — budget rebase time in Phase 2/3.
- **Quorum-semantics mismatch is structural:** engine quorum is fixed count-based N3f1 with a participant set frozen at startup (StaticEpochMonitor) [existingIntegration, eigenlayerDeps], while the on-chain check is stake-weighted BPS against an epoch snapshot that decays. The startup assertion (min-stake (N−f)-subset ≥ threshold_bps, §2a) makes the mismatch *checkable* but only genuinely holds under near-equal weights; a certificate can still clear the engine and fail on-chain when the snapshot rotates mid-flight. Mitigation is the documented assertion + generation checks + restart-on-set-change (same operational restriction the EVM stack already has), not a solved problem.
- **Layer 1 ≠ the whole product:** the gas-killer/service port and the real-workload diff producer are explicitly out of Layer 1 (§1, decision #8). The Phase-4 mapping rule and event ABI are designed so that follow-on effort consumes stable interfaces, but until it ships, "Gas Killer on Solana" means the toolkit + settlement program, not the /trigger-to-settlement product pipeline.
- **Operational carry-overs:** the service#326 resilience bugs (p2p mesh not healing on pod-IP churn, in-memory gate records, infra-error process fatality [transcriptB]) are chain-independent and will recur on the Solana deployment; Jito adds mandatory per-epoch cranking (vault update tracker gates mint/burn [jito]) with no protocol-level cron on Solana (Clockwork dead [solana]) — the keeper is liveness-critical infrastructure we must run.
- **License exposure:** jito-bls-ncn has no license [jito recon] — any verbatim reuse from ck/ncn is blocked until resolved; the plan therefore re-implements its *concepts* (RollingSnapshot weighting, epoch decay) in the Apache-2.0/MIT Unboxed base. Flagged as decision #2/#3.
- **CI claims in commit logs are not evidence:** ck/ncn's "all ci passing" reflects local `act` runs — its workflow cannot build SBF as written [existingIntegration]; Unboxed has zero workflows. The "e2e in CI" bar starts from nothing, as [GH] states. The no-mocks bar now also covers fixture provenance: jito programs are built from pinned source in CI, not trusted as opaque checked-in binaries.
- **UNVERIFIED items:** devnet liveness of the mainnet Jito program IDs and the devnet `epoch_length` value (checked in Phase 0 with Path B as the standing fallback — UNVERIFIED until that check runs); exact CU cost of the full verify path at N=256 operators (jbn's 250k figure is a claim in its code, not a measured recon fact — measured in Phase 4 for D-VERIFY); the exact Agave MAX_CPI_INSTRUCTION_DATA_LEN interaction with the event-helper design at the pinned toolchain version (asserted in Phase 4 tests); whether Cambrian SDK offers reusable components worth depending on (recon calls its docs marketing-heavy — evaluate, don't assume).
