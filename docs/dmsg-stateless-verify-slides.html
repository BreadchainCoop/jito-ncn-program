Jito NCN backend · design decision D-MSG · sign-off before Phase 1

What does the committee sign,
and where does replay protection live?

The stateless verify / D-MSG decision: why the Solana NCN must verify certificates the way EigenLayer's checkSignatures does — as a pure check — and why the replay nonce belongs to each consumer, not to the NCN.

The reference system

How Gas Killer settles on Ethereum today

  • checkSignatures is EigenLayer's view function — the only EigenLayer runtime primitive the SDK consumes.
  • The consumer contract owns everything stateful: the transition nonce, the commitment slot, the events.
The property that matters

Verification is stateless; certificates are facts

  • A certificate states a fact: “quorum signed digest D against stake snapshot S.” Facts don't expire when read — checkSignatures mutates nothing and can be called forever.
  • Replay protection is not the verifier's job. The consumer binds its own transitionIndex inside the digest preimage, so each certificate is usable for exactly one state transition of that consumer.
  • Any number of consumers share one verifier with zero coupling — none of them can invalidate another's pending certificate.
The drafts diverge here

Both Solana drafts made the verifier stateful

Unboxed · CastVote

  • The signed message is the global VoteCounter value.
  • On success: counter += 1 — verification mutates NCN state.

jito-bls-ncn · Vote

  • Message bound to consensus_count; Consensus PDA mutated on success.
  • Same shape: one global, ordered vote stream per NCN.

In both drafts the NCN stopped being a verifier and became a sequencer: every settlement, from every consumer, must pass through one ordered counter.

Problem 1 · concurrency

One counter PDA = one write lock for everyone

  • Solana's whole execution model is per-account locking — a shared writable PDA is the one thing that defeats it.
  • Two consumers settling in the same slot contend; the loser's transaction fails and must rebuild.
Problem 2 · predictability

The digest must be computable before signing

  • Gas Killer's pipeline signs off-chain, in advance: simulate → build diff → compute digest → collect BLS signatures → submit.
  • If the signed message includes “the counter's current value,” the router cannot know that value at signing time — any other consumer's settlement (or a skipped round) shifts it.
  • Result: racing consumers invalidate each other's certificates. On the EVM side this never happens — each consumer knows its own next transitionIndex exactly.
Problem 3 · semantics

Cross-consumer coupling nobody asked for

  • With a global counter, consumer A's settlement changes what consumer B's committee must sign next. That ordering was never part of the protocol.
  • Attestation clarity suffers: did the committee attest “digest D” — or “digest D, provided it lands at counter position k”?
  • The EVM system has no analog of this coupling, so keeping it would be a semantic regression hidden inside “parity.”
The decision · D-MSG

Message = the raw 32-byte digest. Nothing else.

NCN · VerifyCertificate (stateless)

  • hash-to-curve(domain ∥ digest)
  • subtract non-signer G1s from snapshot APK
  • one pairing · signed stake ≥ threshold bps
  • snapshot generation + freshness check
  • writes: none

Consumer · owns replay

  • state PDA holds transition_count — the StateTracker mirror
  • nonce is bound inside the digest preimage:
  • sha256(borsh(idx, state_pda, ix, diff))
  • settle requires idx + 1 == transition_count

The demo VoteCounter survives — demoted to a demo consumer's own state: its CastVote becomes “verify certificate, bump my own counter.”

Why this is safe

NCN-level replay is harmless by construction

  • A certificate that verifies twice is just a fact observed twice. Verification has no side effects to abuse.
  • Settling twice is impossible: after the first settle, transition_count advanced, so the same digest fails idx + 1 == transition_count.
  • The next transition has a new index → new digest → new certificate. One certificate, one transition, per consumer.
  • This is not a new invention — it is the EVM design, ported faithfully: “cert binds only the digest… replay across heights is safe due to on-chain transition-index ordering.”
Putting it together

The Solana settlement flow under D-MSG

  • Consumers settle in parallel — each writes only its own PDA; the NCN snapshot is only ever read.
Why sign-off must precede Phase 1

D-MSG changes two load-bearing things at once

  • The instruction set: stateful CastVote → stateless VerifyCertificate, with the demo counter moved out of the NCN into the example consumer.
  • The signature domain: operators sign the digest, not a counter — so the router scheme crate, the CRYPTO-1 alpha computation, and every differential test vector are built against these exact bytes. Changing it later means re-keying the committee and re-auditing the crypto.
  • Concurrency model becomes per-consumer PDAs — this shapes every account layout Phase 1 touches.
The sibling question · D-VERIFY (not blocking)

CPI into the NCN, or verify inline?

Option A — CPI into VerifyCertificate

  • one audited implementation, shared by all consumers
  • costs CPI depth + invoke overhead

Option B — inline verify in settlement

  • reads the Snapshot PDA directly, cheaper
  • duplicates the verification code → larger audit surface

Because D-MSG made both options stateless, this is purely a compute-unit / audit-surface trade with no protocol consequence — it gets decided by measurement in Phase 4, and nothing upstream waits on it.

Sign-off

D-MSG, as proposed

  • SIGN committee message = raw 32-byte digest, domain-separated at hash-to-curve
  • VERIFY NCN VerifyCertificate is stateless — generation-checked, threshold-checked, zero writes
  • REPLAY nonce is consumer-local (transition_count in each state PDA), bound inside the digest preimage
  • REJECT NCN-global vote counters in the verify path (both drafts) — lock contention, unpredictable digests, cross-consumer coupling

Approving this fixes the Phase 1 instruction set and the signature domain together — the last blocking design input before the build starts.
